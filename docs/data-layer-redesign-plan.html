Data Layer Redesign Plan

Draft — for review before any code changes

1. Problem

The financial tables model the same real-world concept (“money we owe a supplier”) twice, and several names don’t describe what the table holds.

Current tables (financial core)

TableFileReality
expensecore/expenses/schema.tsBill/receipt header. No amounts, only payment_status.
expense_linecore/expenses/schema.tsItem on a bill: description, qty, unit_rate_cents, amount_cents, charged to an allocation.
payablecore/payments/schema.tsA second, competing “bill” with its own title, amount_cents, status.
ledger_paymentcore/payments/schema.tsA payment. Points at both expense_id (no FK) and payable_id.
allocationcore/allocations/schema.tsActually a project budget category (name + budget_cents).

Specific issues

  1. expense vs payable duplication. createPayableExpense in core/expenses/service.ts writes expense rows but returns payable-shaped responses. ledger_payment hedges with two nullable parent columns. Every consumer must know which model a given bill lives in.
  2. allocation is misleading. It’s a budget line of a project, not a payment allocation.
  3. Missing FKs: ledger_payment.expense_id, expense.receipt_file_id, project_attachment.project_id, member_project.project_id, project.archived_by.
  4. Free-text status columns (expense.payment_status, payable.status, project.status, etc.) documented only in comments — no pg enums or check constraints.
  5. Derived data on headers: expense.payment_status is derivable from payments vs line totals; expense has no stored total, forcing consumers to sum lines.

2. Target model

One bill concept. Mental model:

organization
  └── project
        └── budget_category   (renamed from allocation)
              ▲
expense ──── expense_item     (renamed from expense_line)
   ▲
payment                        (renamed from ledger_payment)

Table changes

CurrentNewNotes
allocationbudget_categorySame columns.
expenseexpense (keep)Add due_date, currency, total_cents, description; keep payment_status as pg enum unpaid | partial | paid. FK on receipt_file_id → uploaded_file.
expense_lineexpense_itemRename allocation_id → budget_category_id, unit_rate_cents → unit_cents.
payabledroppedExisting payables migrate to expense rows (title → description, amount → single expense_item, due_date carried over).
ledger_paymentpaymentDrop payable_id; make expense_id NOT NULL with FK. Keep idempotency unique index.

Enums (pg enums via drizzle pgEnum)

  • expense_payment_status: unpaid, partial, paid
  • project_status: active, completed, on_hold
  • file_purpose: attachment, receipt

3. Migration steps (single drizzle migration)

  1. Rename allocation → budget_category; expense_line → expense_item (+ column renames).
  2. Alter expense: add due_date, currency (default UGX), total_cents, description.
  3. Data move: for each payable, insert an expense (+ one expense_item against a designated/first budget category of the project) and repoint its ledger_payment.payable_id rows to the new expense_id.
  4. Rename ledger_payment → payment; drop payable_id; add FK + NOT NULL on expense_id.
  5. Drop payable.
  6. Backfill expense.total_cents from item sums; recompute payment_status.
  7. Add missing FKs (receipt_file_id, project_attachment.project_id, member_project.project_id).
  8. Convert status varchars to pg enums.

If there is no production data yet, steps 3 and 6 collapse and we can instead regenerate a fresh initial migration.

4. Code impact

  • Schemas: core/expenses/schema.ts, core/payments/schema.ts, core/allocations/schema.ts (folder rename → core/budget-categories/), core/shared/schema.ts.
  • Services/repos: core/expenses/{service,repository}.ts (remove payable branches, delete createPayableExpense duality), core/payments/{service,repository}.ts, core/projects/service.ts.
  • Tests: core/payments/service.test.ts, core/projects/service.test.ts.
  • App layer: app/admin/expenses/**, app/admin/payments/actions.ts, app/admin/projects/** (incl. new/allocation/page.tsx → rename route to new/budget/).

5. Open questions

  1. Is there production data, or can we regenerate migrations from scratch?
  2. Should expense.total_cents be stored (denormalized, updated in the write transaction) or always computed? Recommendation: stored, since all writes already go through one service transaction.
  3. Keep the core/allocations folder name or rename to core/budget-categories? Recommendation: rename for consistency.
  4. UI wording: “Budget category” vs “Budget line” vs “Cost category”?
